Responsive layouts don't have to be a struggle

But they will be at first. In any skill, hobby, or profession, developing good technique takes knowledge and practise.

Once you have learned some things about them, though, responsive layouts are actually quite pleasant to make (and, of course, to use).

Over the years, HTML5 and CSS have matured into very capable languages for creating and styling web interfaces. You should (hopefully) feel at the end of this course the ability to create (at least approximately) any web layout you like using these languages.

Sadly, the button above can't help you. Inspecting the source code of this document, however, can.

I have tried to summarize everything we have learned (and some things we haven't) about HTML and CSS best practises into the design and documentation of this page.

Inspect the elements, read the comments, and feel free to re-use this code in your future projects. Let me know if you have any questions!

Mobile-first

Mobile layouts are typically single-column, perfectly suited to the block stacking nature of unstyled HTML. Mobile-first designs force the developer to emphasize coherent structure and markup.

Inspect this page in a mobile layout, both with and without page style. Then, view the source CSS. What, if anything, does the CSS add to the layout of this page?

Responsive

Responsive web designs have well-defined layouts for each device used to browse the page. Tablet and Destkop layouts are often composed of many columns to take advantage of wider viewport space.

Use responsive mode in your browser's developer tools to inspect this page at a variety of widths. How does the overall layout change at each breakpoint? How are the styles of some components changed?

Accessible

Accessible design ensures that websites are usable by as many people as possible. Page semantics, colour contrast, keyboard interactivity, network speed, and device capability are just some properties that developers should have in mind when designing websites.

Use the accessibility auditing tool to inspect these properties on this page. What has been done well? What has room for improvement?

It really doesn't have to be so hard

With practise, you will obtain genuine fluency in expressing aesthetic and responsive interfaces using the modern tools of the trade.

Remember that when you are developing websites, or any programs, you are not the only person who uses them. In fact, if you are successful in this field, your code and your projects will be used and read many orders of magnitude more times than the amount of time you spend creating them.

Importantly, accessibility also means understanding that there are many other kinds of users who have accessibility needs.

Instead of designing for your own use, then patching until you achieve semi-functionality for anyone else, a wise developer places themselves into the shoes of the person they’re designing for and builds something which speaks their design language. For visually impaired users, this might mean laying out information in a more logical sense than in a spatial sense.

For instance, consider someone who cannot afford a computer as nice as the one you are using. A website with a huge JavaScript bundle, unoptimized images, and always-on network requirements is not accessible to users who are on low-bandwidth mobile connections or those that have a data cap — you’re essentially charging poorer users a tax to use your website.

Localization is another kind of accessibility, Users in different locales speak not only different natural languages, but different design languages. Users of right-to-left languages like Arabic don’t just reverse their strings but also the entire layout of the page. Chinese and Japanese users are more familiar with denser UIs than the typical Western user. And subtitles and transcripts are important for Deaf users, but also useful for users who are consuming your content in a second language.

Intuitiveness is another important detail. Not everyone understands what your icons mean, for a start. They may not have the motor skill to hold their mouse over the button and read the tool-tip, either, and might not know that they can do that in the first place! Reliance on unfamiliar design language in general is a kind of inaccessible design. Flashing banner ads are also inaccessible for users with ADHD, and if we’re being honest, for everyone else, too. Software which is not responsive on many kinds of devices (touch, mouse and keyboard, different screen sizes, aspect ratios, orientations) is not accessible.

Adequate answers to these problems are **relatively easy to achieve if we use the tools made available to us well.** We do not have to copy the anti-patterns of social-media monsters like Facebook, Twitter, Instagram, and the rest. Designs which are deliberately addictive are not accessible.

It is a fool’s errand to aim for perfect accessibility for all users, but we need to understand that our design choices are excluding people from using our tools. We need to design our software with accessibility in mind from the ground up, and with a broad understanding of accessibility that acknowledges that simple, intuitive software is the foundation of accessibility which works for everyone, including you and me — and not retroactively adding half-assed tools to fundamentally unusable software.

Break it down

Not unlike lego, HTML structural elements are blocks by default. Each element is a unit on its own that, when combined with other elements, comprises a greater whole.

When designing, ask yourself: what IS each element on my page? What are its constraints? How does it relate to the elements around it?

Work your way up

Not unlike cooking, CSS styles are recipes for combining and transforming ingredients into a finished meal website.

When cooking adding css, ask yourself: does this style help fulfill a requirement of my project? Try to avoid band-aid solutions.

~*~

Not unlike any other programming language, it's helpful to debug as you develop. Temporary borders around elements in your CSS can help visualize how that element is flowing in your document.

Finally: there's no need to be so serious all the time. It's okay to make mistakes. The important things it to make stuff so that you can learn how it works.
