Responsive layouts don't have to be a struggle...

... but they will be at first. In any skill, hobby, or profession, developing good technique takes knowledge and practise.

Once you have learned some things about them, though, responsive layouts are actually quite pleasant to make (and, of course, to use).

Over the years, HTML5 and CSS have matured into very capable languages for creating and styling web interfaces. You should (hopefully) feel at the end of this course the ability to create (at least approximately) any web layout you like using these languages.

Sadly, the button above can't help you. Inspecting the source code of this document, however, can.

I have tried to summarize what we have learned (and some things we haven't) about HTML and CSS best practises into the design and documentation of this page.

Inspect the elements, read the source code comments, and feel free to re-use this code in your future projects. Let me know if you have any questions!

Mobile-first

Mobile layouts are typically single-column, perfectly suited to the block stacking nature of unstyled HTML. Mobile-first designs force the developer to emphasize coherent structure and markup.

Inspect this page in a mobile layout, both with and without page style. Then, view the source CSS for this website. What, if anything, does the CSS add to the layout of this page?

Responsive

Responsive web designs have well-defined layouts for each device used to browse the page. Tablet and Destkop layouts are often composed of many columns to take advantage of wider viewport space.

Use responsive mode in your browser's developer tools to inspect this page at a variety of widths. How does the overall layout change at each breakpoint? How are the styles of some components changed?

Accessible

Accessible design ensures that websites are usable by as many people as possible. Page semantics, colour contrast, keyboard interactivity, network speed, and device capability are just some properties that developers should have in mind when designing websites.

Use the accessibility auditing tool to inspect these properties on this page. What has been done well? What has room for improvement?

It really doesn't have to be so hard

With practise, you will obtain genuine fluency in expressing aesthetic and responsive interfaces using the modern tools of the trade.

Remember that when you are developing websites, or any programs, you are not the only person who uses them. In fact, if you are successful in this field, your code and your projects will be used and read by others many orders of magnitude more than the amount of time spent creating them.

While the content of your websites and your projects is of course important, the accessibility of your website is the key factor that determines whether that content can be used and understood by others. Therefore, it is important to understand the main obstactles to accessibility in user interface design.

Break it down

Not unlike lego, HTML structural elements are blocks by default. Each element is a unit on its own that, when combined with other elements, comprises a greater whole.

When designing, ask yourself: what IS each element on my page? What are its constraints? How does it relate to the elements around it?

Work your way up

Not unlike cooking, CSS styles are recipes for combining and transforming ingredients into a finished meal website.

When cooking adding css, ask yourself: does this style help fulfill a requirement of my project? Try to avoid band-aid solutions.

~*~

Not unlike any other programming language, it's helpful to debug as you develop. Temporary borders around elements in your CSS is one way to help visualize how that element is flowing in your document.

Finally: there's no need to be so serious all the time. It's okay to make mistakes. The important things it to just make stuff so that you can learn how it works.
